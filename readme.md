# 법령 검색·지식그래프 미니 프로젝트

**와이매틱스(Ymatics) 인턴십**에서 진행하는 미니 프로젝트입니다.

112 시스템의 법령 검색·지식그래프 파이프라인을, 공개 데이터와 소형 모델로 재현합니다.  
미니 시스템에서 정확도를 숫자로 검증한 뒤, **개선안을 정리해 실제 112 시스템에 반영·제안**하는 것이 최종 목표입니다.

> 보안상 실제 112/경찰청 소스·서버에는 접근하지 않습니다.  
> 실험은 개인 노트북의 미니 시스템에서만 수행하고, 검증된 결과는 제안서 형태로 전달합니다.

---

## 배경

112 업무에서는 관련 법령을 빠르게·정확히 찾는 것이 중요합니다.  
벡터 검색(RAG)만으로는 조문 구조·개정·교차 참조를 다루기 어렵고, 지식그래프(KG)를 함께 쓰면 관계가 명확해질 수 있습니다.

이 프로젝트는 그 가설을 **작은 규모로 직접 구현·측정**합니다.

---

## 목표

1. 법령이 지식그래프로 잘 구축되는가?
2. 파싱(PDF → 텍스트)이 얼마나 안정적인가?
3. 비개발자가 개정 법령을 추가해도 그래프·검색 인덱스가 꼬이지 않는가?
4. 벡터 검색만 쓸 때와 **KG를 함께 쓸 때**, 정답률이 얼마나 달라지는가?

모든 개선은 **바꾸기 전/후 정답률**로 비교합니다.

---

## 시스템 개요

```
법령 문서 (law.go.kr)
    → 파싱 / 청킹
    → 임베딩 → 벡터 검색
    → (지식그래프 추출·조회)
    → LLM 답변 (RAG)
    → 정답 질문 세트로 정확도 측정
```

| 단계 | 설명 |
|------|------|
| 파싱 | PDF·웹 텍스트에서 조문 분리, 표·개정표시·조/항/호/목 구조 유지 |
| 청킹 | 조문 또는 항 단위로 분할 |
| 임베딩 · 검색 | 의미 유사도로 관련 조문 검색 |
| 지식그래프 | 개체·관계 추출, 정규화·중복 병합, 개정 시 현행화 |
| RAG | 검색된 조문을 바탕으로 LLM이 답변 생성 |
| 측정 | 정답 질문 세트·RAGAS 등으로 정답률 산출 |

---

## Tech Stack

| 구분 | 기술 |
|------|------|
| 언어 | Python |
| 임베딩 | Hugging Face `sentence-transformers` (`jhgan/ko-sroberta-multitask`, `intfloat/multilingual-e5-small`) |
| LLM | Ollama 로컬 모델 (`qwen2.5:3b`, `gemma2:2b` 등) |
| RAG | 벡터 검색 + 로컬 LLM 답변 생성 |
| 지식그래프 | LLM 기반 개체·관계 추출, 그래프 시각화·흠집 감사 |
| 파싱 | `pdfplumber` 등 (PDF 구조 관찰용) |
| 측정 | 정답 질문 세트 기반 키워드/정답률, RAGAS |
| 데이터 | [국가법령정보센터](https://www.law.go.kr) 공개 법령 |

노트북 CPU만으로 동작하도록 설계하며, 조문 규모는 10–20개 수준을 유지합니다.

---

## 샘플 데이터

| 법령 | 선정 이유 |
|------|-----------|
| 경범죄처벌법 | 조문 구조가 단순해 워밍업에 적합 |
| 도로교통법 (음주·무면허) | 112 업무와 가깝고 질문 구성이 용이 |
| 형법 (폭행·협박) | 주제가 다른 법을 섞어 그래프 분리 양상 관찰 |

---

## 실무 반영 방식

1. 미니 시스템에서 가설을 구현하고 정답률로 검증한다.
2. before / after 수치와 개선안을 제안서로 정리한다.
3. 실제 112 시스템 반영은 회사 담당자(멘토)를 통해 검토·적용한다.

---

## 원칙

- **측정 없는 개선은 없다** — 전/후 정답률을 숫자로 남긴다.
- **미니 시스템에서만 실험** — 실서비스·비공개 코드에 접근하지 않는다.
- **재현 가능한 결과** — 공개 데이터·공개 모델로 누구나 같은 실험을 따라갈 수 있게 한다.
